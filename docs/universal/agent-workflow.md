# エージェントワークフロー設計

## マルチエージェント構成

SDD プロジェクトでは、3つのエージェント（ツール）が役割分担する。

### 役割分担

| エージェント | 役割 | 主な操作 |
|---|---|---|
| Cursor（IDE） | 仕様書の編集・コードレビュー | `spec/api/*.yaml` の編集 |
| Claude Code（ターミナル） | コード生成・テスト・Git 操作 | コード生成、テスト実行、コミット、プッシュ |
| GitHub Actions（CI/CD） | 自動テスト・自動デプロイ | pytest 実行、sam deploy |

### ワークフロー

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────────┐
│   Cursor      │     │   Claude Code     │     │ GitHub Actions  │
│  (仕様書編集)  │     │ (コード生成/Git)   │     │   (CI/CD)       │
└──────┬───────┘     └────────┬─────────┘     └────────┬────────┘
       │                      │                         │
       │ 1. spec/*.yaml 編集  │                         │
       │─────────────────────▶│                         │
       │                      │ 2. コード生成            │
       │                      │ 3. テスト生成            │
       │                      │ 4. テスト実行            │
       │                      │ 5. git commit           │
       │                      │ 6. git push             │
       │                      │────────────────────────▶│
       │                      │                         │ 7. pytest
       │                      │                         │ 8. sam deploy
       │                      │                         │ 9. 結果通知
       │                      │◀────────────────────────│
       │                      │                         │
```

## Claude Code のスキル体系

### 汎用 Skills（`.claude/skills/common/`）

どのプロジェクトでもそのまま使える共通ルール：

| スキル | トリガー | 内容 |
|---|---|---|
| `git-workflow` | 「コミットして」「プッシュして」 | Git 操作の標準ルール |
| `error-handling` | 「エラーを解析して」「原因を調べて」 | エラー修正ワークフロー |
| `test-workflow` | 「テストして」「テストを実行して」 | テスト実行・結果報告 |
| `doc-writer` | 「ドキュメントを書いて」「READMEを更新して」 | ドキュメント生成ルール |

### プロジェクト固有 Skills（`.claude/skills/project/`）

プロジェクトの技術スタックに依存するルール：

| スキル | トリガー | 内容 |
|---|---|---|
| `python-lambda` | `src/` のコードを扱う時 | Lambda ハンドラの構造・規約 |
| `sam-architect` | `template.yaml` を扱う時 | SAM テンプレートの設計原則 |

## エラー解析・修正ワークフローとの統合

CI/CD パイプラインでエラーが発生した場合のエージェントの役割分担：

### Claude（ブラウザ）
- エラーの概要を説明
- Claude Code へのプロンプトを設計

### Claude Code（ターミナル）
- CloudWatch ログの取得・解析
- エラーの原因特定
- ユーザーへの報告・選択肢の提示
- 選択に基づいて修正を実行

### Cursor（IDE）
- 修正内容のレビュー
- 必要に応じて手動修正

### エラー修正フロー

```
GitHub Actions 失敗
  │
  ▼
Claude（ブラウザ）でエラー概要を確認
  │
  ▼
Claude Code に「エラーを解析して」と依頼
  │
  ▼
Claude Code が error-handling Skill に従い：
  1. ログ取得・原因分析
  2. ユーザーに報告・選択肢提示
  3. 選択に基づいて修正実行
  4. テスト再実行
  │
  ▼
修正をコミット・プッシュ
  │
  ▼
GitHub Actions が再実行
```

## Human Gate の設計

以下の操作は自動実行せず、必ずユーザーの承認を得る：

| 操作 | 理由 |
|---|---|
| 本番環境へのデプロイ | Yellow Zone |
| API インターフェースの破壊的変更 | Yellow Zone |
| DynamoDB テーブル定義の変更 | Yellow Zone |
| IAM ポリシーの変更 | Yellow Zone |
| 3回以上のエラー修正ループ | エスカレーション条件 |

## 質問ドキュメントによる意思決定フロー

### 概要

複数の質問が相互に関連する場合や、全体的な方向性を決める必要がある場合、コンソールでの一問一答ではなく「質問ドキュメント」を使用する。

### コンソール質問 vs 質問ドキュメントの使い分け

| 状況 | 方法 |
|---|---|
| Yes/No の単一質問 | コンソールで直接質問 |
| 明確な二択・三択 | コンソールで直接質問 |
| 複数の質問が相互に関連 | 質問ドキュメント |
| 前提条件の整理が必要 | 質問ドキュメント |
| 設計判断・方向性の決定 | 質問ドキュメント |
| 決定事項を記録に残したい | 質問ドキュメント |

### フロー図

```
Claude Code が複雑な判断を検知
  │
  ▼
docs/questions/YYYY-MM-DD_<トピック>.md を作成
  ├─ 背景の説明
  ├─ 質問一覧（選択肢付き）
  └─ 回答欄（ユーザーが記入）
  │
  ▼
コンソールで「質問ドキュメントを作成しました」と通知
  │
  ▼
ユーザーがドキュメント内の回答欄に記入
  │
  ▼
ユーザーが「回答した」と通知
  │
  ▼
Claude Code が回答を読み取り、作業を実行
  │
  ▼
質問ドキュメントに決定事項・実行結果を追記
```

### メリット

- **全体像の把握**: ユーザーが全質問を俯瞰して判断できる
- **前提条件の共有**: 質問の背景や依存関係が明確になる
- **決定の記録**: 後から「なぜこの判断をしたか」を振り返れる
- **非同期対応**: ユーザーが回答を考える時間を確保できる

## ユーザーとのインタラクション設計

### 原則

Claude Code とユーザーのやり取りは以下の 2 つのモードで行う：

**モード 1：一問一答**
シンプルな質問・確認に使用する。コンソール上でやり取りを完結させる。

**モード 2：質問ドキュメント**
複雑な判断・重要な決定に使用する。ファイルを介してやり取りを行い、記録を残す。

### モードの選択基準

```
質問や判断が必要
  ↓
質問数は？
  ├─ 1〜2個 → 一問一答
  └─ 3個以上 → 質問ドキュメント

質問間に依存関係がある？
  ├─ なし → 一問一答
  └─ あり → 質問ドキュメント

記録として残す必要がある？
  ├─ なし → 一問一答
  └─ あり → 質問ドキュメント
```

### 質問ドキュメントのライフサイクル

```
作成（Claude Code）
  ↓
回答記入（Human / Cursor）
  ↓
実行確認（Claude Code → Human に確認）
  ↓
実行（Claude Code）
  ↓
結果記録（Claude Code）
  ↓
コミット（Claude Code）
  ↓
保管（.claude/questions/）
```

詳細な仕様は `docs/universal/question-document-spec.md` を参照。
