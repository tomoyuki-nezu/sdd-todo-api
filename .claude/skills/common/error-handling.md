# Error Handling Workflow

## 基本方針
エラーが発生した場合、以下のワークフローに従うこと。
ただし自動修正を実行する前に、必ず最初の1回は
ユーザーに状況を報告し、対処方法の選択を求めること。

## ワークフロー

### Step 1：エラーの収集と分析
- エラーメッセージ・スタックトレースを収集
- ログファイル・コンソール出力を確認
- 関連するソースコードを確認
- エラーの種類を分類：
  - 設定ミス（環境変数・認証情報・エンドポイント）
  - コードのバグ（ロジックエラー・型エラー）
  - インフラの問題（リソース未作成・権限不足）
  - 依存関係の問題（パッケージバージョン・互換性）

### Step 2：ユーザーへの報告（必須・自動修正前に必ず実施）

以下の形式でユーザーに報告すること：

```
## エラー報告

### 発生したエラー
（エラーメッセージを簡潔に記載）

### 推測される原因
1. （最も可能性が高い原因）
2. （次に可能性が高い原因）
3. （その他の可能性）

### 対処方法の選択肢
A. （推奨）（具体的な修正内容）
B. （代替案）（具体的な修正内容）
C. 詳細をさらに調査する
D. 手動で対応する（Claude Code は修正しない）

どの方法で進めますか？（A/B/C/D）
```

### Step 3：ユーザーの選択に基づいて実行

ユーザーが選択した方法に従って実行する。
「A」または「B」を選択した場合は以下を実施：

1. 修正を適用する
2. テストを再実行して修正が有効か確認する
3. 修正が無効だった場合は Step 1 に戻る
4. 自動修正を最大 3 回試みて解決しない場合は
   ユーザーにエスカレーションする

### Step 4：修正完了の報告

修正が完了したら以下を報告する：
- 適用した修正内容
- 修正後のテスト結果
- 再発防止のための推奨事項（必要な場合）

## 自動修正ループの制限
- 最大試行回数: 3 回
- 3 回で解決しない場合は必ずユーザーにエスカレーション
- 本番環境への影響がある修正は必ず事前確認

## エスカレーション条件（即座にユーザーに確認）
- Red Zone の操作が必要な場合
- 破壊的変更（API インターフェース変更など）が必要な場合
- 原因が特定できない場合
- セキュリティ上の問題が検出された場合
