# Error Handling Workflow

## 基本方針
エラーが発生した場合、以下のワークフローに従うこと。
ただし自動修正を実行する前に、必ず最初の1回は
ユーザーに状況を報告し、対処方法の選択を求めること。

## ワークフロー

### Step 1：エラーの収集と分析
- エラーメッセージ・スタックトレースを収集
- ログファイル・コンソール出力を確認
- 関連するソースコードを確認
- エラーの種類を分類：
  - 設定ミス（環境変数・認証情報・エンドポイント）
  - コードのバグ（ロジックエラー・型エラー）
  - インフラの問題（リソース未作成・権限不足）
  - 依存関係の問題（パッケージバージョン・互換性）

### Step 2：報告方法の選択

エラーの複雑さに応じて報告方法を選択する：

#### シンプルなエラーの場合（一問一答）

原因が明確で選択肢が 1〜2 個の場合はコンソール上で直接報告する：

```
エラーが発生しました。
原因: ○○
修正方法: ○○
実行してよいですか？（yes/no）
```

#### 複雑なエラーの場合（質問ドキュメント）

以下のいずれかに該当する場合は質問ドキュメントを作成する：
- 原因が複数考えられる
- 選択肢が 3 つ以上ある
- 修正方法に前提条件がある
- 本番環境への影響がある

質問ドキュメントのフォーマットは
`docs/universal/question-document-spec.md` に従うこと。

### Step 3：ユーザーの選択に基づいて実行

ユーザーが選択した方法に従って実行する。
「A」または「B」を選択した場合は以下を実施：

1. 修正を適用する
2. テストを再実行して修正が有効か確認する
3. 修正が無効だった場合は Step 1 に戻る
4. 自動修正を最大 3 回試みて解決しない場合は
   ユーザーにエスカレーションする

### Step 4：修正完了の報告

修正が完了したら以下を報告する：
- 適用した修正内容
- 修正後のテスト結果
- 再発防止のための推奨事項（必要な場合）

## 自動修正ループの制限
- 最大試行回数: 3 回
- 3 回で解決しない場合は必ずユーザーにエスカレーション
- 本番環境への影響がある修正は必ず事前確認

## エスカレーション条件（即座にユーザーに確認）
- Red Zone の操作が必要な場合
- 破壊的変更（API インターフェース変更など）が必要な場合
- 原因が特定できない場合
- セキュリティ上の問題が検出された場合

## 質問ドキュメントへのフォールバック

エラーの原因が複数考えられ、修正方針の決定に
複数の相互関連する質問が必要な場合は、
コンソールでの一問一答ではなく質問ドキュメントを作成すること。

→ 詳細は `question-doc` Skill（`.claude/skills/question-doc/SKILL.md`）を参照
