# エラー解析・修正ワークフロー

## 設計思想

Claude Code によるエラー修正を自動化しつつ、以下の原則を維持する：

1. **最初の1回は必ずユーザーに報告・確認する**
2. **選択肢を提示してユーザーが判断する**
3. **自動修正は最大3回まで**
4. **解決しない場合は必ずエスカレーション**

## ワークフロー図

```
エラー発生
  │
  ▼
Claude Code がエラーを収集・分析
  │
  ▼
ユーザーへの報告（必須）
  ├─ 発生したエラーの説明
  ├─ 推測される原因（複数）
  └─ 対処方法の選択肢（A/B/C/D）
  │
  ▼
ユーザーが選択
  │
  ├─ A or B → Claude Code が自動修正
  │              │
  │              ├─ 成功 → 修正完了レポート
  │              └─ 失敗 → 再試行（最大3回）
  │                          └─ 3回失敗 → エスカレーション
  │
  ├─ C → さらに詳細調査
  └─ D → 手動対応（Claude Code は修正しない）
```

## エラーの分類

| 分類 | 例 | 対処の目安 |
|---|---|---|
| 設定ミス | 環境変数未設定、認証情報不足 | 設定ファイルの修正で解決 |
| コードのバグ | ロジックエラー、型エラー | ソースコードの修正 |
| インフラの問題 | リソース未作成、権限不足 | AWS リソースの設定 |
| 依存関係の問題 | パッケージバージョン不整合 | requirements.txt の更新 |

## Human Gate（必ずユーザーに確認する条件）

以下の場合は自動修正を行わず、必ずユーザーに確認する：

- 本番環境への影響がある修正
- API インターフェースの変更
- セキュリティ上の問題
- 原因が特定できない場合
- 3回の自動修正で解決しない場合
- Red Zone の操作が必要な場合（spec/constitution.md 参照）

## Claude Code へのトリガーフレーズ

以下のフレーズで error-handling Skill が起動する：

- 「エラーを解析して」
- 「原因を調べて」
- 「テストが失敗しているので修正して」
- 「ログを確認して」

## レポート形式

### エラー報告（修正前）

```
## エラー報告

### 発生したエラー
（エラーメッセージを簡潔に記載）

### 推測される原因
1. （最も可能性が高い原因）
2. （次に可能性が高い原因）

### 対処方法の選択肢
A. （推奨）（具体的な修正内容）
B. （代替案）
C. 詳細をさらに調査する
D. 手動で対応する

どの方法で進めますか？（A/B/C/D）
```

### 修正完了報告

```
## 修正完了

### 適用した修正
（何をどう変更したか）

### テスト結果
（修正後の実行結果）

### 再発防止の推奨事項
（必要な場合）
```
